# 订单系统测试框架

> **任何功能 = 场景 + 输入 + 输出 + 角色 + 状态 + 异常**

------

#  一、业务场景分析（Scenario）

> **目的：确保不会漏掉用户真实的使用路径**

问自己：

- 用户通常会怎么用这个功能？
- 用户会从哪些入口触发？
- 使用流程有几个步骤？

输出：

+ 业务流程图
+ 所有入口场景
+ 用户使用路径

------

# 二、输入分析（Input）

> **目的：确保输入导致的所有分支逻辑都被覆盖**

输入包括：

### 1. 正常输入

- 符合规则的输入

### 2. 非法输入

- 空输入
- 超长输入
- 格式不对
- 特殊字符、emoji、敏感词等

### 3. 边界输入

- 最大值/最小值
- 库存为 0
- 数字溢出
- 列表第一项、最后一项

输出：

+ 输入清单
+ 每个输入的预期行为

------

# 三、输出分析（Output）

> **目的：验证“结果是否完全正确”**

输出包括：

### 1. UI 输出

- 显示是否更新
- 金额是否正确
- 列表是否刷新

### 2. 后端输出

- 数据库是否写入
- 关联数据是否生成（订单、评论记录、日志）

### 3. 业务输出

- 订单状态是否正确
- 库存是否扣除
- 评论点赞数量是否正确

输出：

+ UI 层验证点
+  数据层验证点
+ 业务结果验证点

------

# 四、角色分析（Role）

> **目的：每种用户权限不同 → 触发的逻辑不同**

典型角色：

- 普通用户
- 服务人员
- 管理员
- 未登录用户
- 黑名单用户

你要做一个矩阵：

> **角色 × 功能 = 所有组合都必须测试**

输出：
 ✔ 角色权限矩阵
 ✔ 每个角色允许/禁止的操作

------

# 五、状态分析（State）

> **目的：任何状态变化都需要测试，不然必漏点**

功能相关状态包括：

### 1. 业务数据状态

- 商品（有库存/无库存/下架/限购）
- 动态（正常/被删除/审核失败）
- 优惠券（已领/未领/过期/已用）
- 订单（）

### 2. 用户状态

- 黑名单
- 未实名
- 未登录

### 3. 系统状态

- 自提点关闭
- 配送不可用
- 评论关闭
- 商品不能购买

做矩阵：

> **状态 × 操作 = 全覆盖**

输出：
 ✔ 状态矩阵
 ✔ 各状态下的期望行为

------

# 六、异常分析（Exception）

> **目的：系统异常是线上最常见问题，需要主动测**

异常包括：

### 1. 网络异常

- 提交时断网
- 弱网、超时

### 2. 接口异常

- 401（未登录）
- 403（权限不足）
- 429（频率限制）
- 500（服务器错误）

### 3. 并发异常

- 同时点击多次
- 多人同时购买最后一件库存

### 4. 不可控异常

- 第三方支付失败
- 第三方服务超时

输出：
 ✔ 异常路径
 ✔ 恢复策略
 ✔ 重试是否安全

------

# 通用测试框架 — 一页总表

| 序号 | 测试维度 | 要做什么                       |
| ---- | -------- | ------------------------------ |
| 1    | 业务场景 | 列所有入口、所有流程、所有路径 |
| 2    | 输入     | 正常 + 非法 + 边界输入全覆盖   |
| 3    | 输出     | UI、数据、业务结果 3 层验证    |
| 4    | 角色     | 角色 × 功能 权限矩阵           |
| 5    | 状态     | 状态 × 操作 组合矩阵           |
| 6    | 异常     | 网络/接口/并发/第三方异常      |

这是你可以复用多年的“万能框架”。

------

# 🧩 如何使用这个框架？

只要你拿到新功能，就做这 3 步：

### **步骤 1：按六维度分析功能 → 写出测试点**

### **步骤 2：把测试点组织成测试用例表**

### **步骤 3：补充异常和边界场景 → 保证不漏测**

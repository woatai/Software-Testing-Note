## 这份笔记怎么用

这份是面试前冲刺版，只保留最容易被问、最适合结合我自己的框架 `E:\pytest_auto_api_my` 回答的问题。

项目定位要说准：

> 我自己基于 `pytest + requests + yaml` 搭建了一个接口自动化框架，用 CRMEB 商城接口做练习场景，主要覆盖登录、商品、购物车、地址、下单这些主流程。这个项目重点不是覆盖所有业务，而是验证我对接口自动化框架分层、数据驱动、接口关联、断言和失败定位的理解。

## 回答主线

遇到接口自动化框架题，不要散着背工具名，按这个顺序讲：

1. 目标：减少重复手工调接口，沉淀稳定回归用例。
2. 技术栈：`pytest + requests + yaml`。
3. 分层：YAML 用例、pytest 执行、请求封装、断言、提取、上下文、配置。
4. 链路：登录提取 token，商品提取商品 ID，购物车和下单接口引用上下文变量。
5. 定位：看请求参数、响应体、断言失败点、上下文变量，必要时结合 SQL 查库。
6. 不足：数据库断言、数据清理、失败重试、报告展示还可以继续增强。

## 我的框架怎么讲

可以直接说：

> 我的框架按职责分层。`data/` 放 YAML 用例数据，`test_case/` 放 pytest 测试入口，`util/requestsUtils/` 做 requests 请求封装，`util/assertion/` 做断言，`util/extract/` 做响应字段提取，`util/context/` 管理上下文，`common/` 放环境和路径配置。这样接口数据和测试代码分离，后续新增接口时主要维护 YAML，不需要每个测试脚本里重复写请求、断言和提取逻辑。

执行流程：

```text
读取 YAML 用例
    ↓
合并公共参数
    ↓
替换上下文变量
    ↓
统一发送 HTTP 请求
    ↓
执行状态码 / JSONPath 断言
    ↓
提取响应字段到上下文
    ↓
后续接口继续引用
```

### 2. 一条接口自动化用例是怎么执行的？

思路：

- 替换上下文变量。
- 发送 HTTP 请求。

回答：

### 4. 多接口依赖怎么处理？

思路：

- 先说接口依赖本质：上一个接口返回值给下一个接口用。
- 再说解决方式：提取变量 + 上下文保存 + 占位符引用。
- 最后结合 token、商品 ID、订单号。

回答：

> 我用上下文管理器处理接口依赖。比如登录接口返回 token 后，把 token 提取到上下文；商品接口返回商品 ID 后，也提取到上下文。后面的购物车、确认订单、创建订单接口，就可以通过占位符引用这些值。结合我的下单主流程，最典型的依赖就是 token、product_id、cart_id、address_id、order_id，这些都不手动复制，而是让上游接口提取、下游接口引用。

### 5. 断言怎么设计？

思路：

- 不只看 HTTP 200。
- 分三层：状态码、业务码/结构、关键业务字段。
- 结合登录 token、下单订单号。

回答：

> 我一般分三层断言。第一层是 HTTP 状态码，比如是不是 200；第二层是业务码和返回结构，比如 code、message、data 是否符合预期；第三层是关键业务字段，比如登录是否返回 token、下单是否返回订单号。结合我的项目，登录接口不能只看请求成功，还要判断 token 是否返回；下单接口也不能只看返回成功，还要判断订单号是否返回。

### 6. 写自动化过程中遇到的问题及挑战，如何解决？

思路：

- 接口依赖：用上下文解决。
- 数据不稳定：固定测试数据，后续补数据清理。
- 失败定位：保留请求和响应调试信息。

回答：

> 我遇到比较典型的问题是接口依赖、测试数据不稳定、失败定位不清楚。接口依赖方面，我用上下文管理 token、商品 ID、订单号；测试数据方面，尽量选择稳定数据，把关键数据放到 YAML 或配置里；失败定位方面，请求封装里会保留 method、url、headers、params、body、响应体等信息。比如下单主流程失败时，我会先检查上下文里有没有 token、product_id、cart_id、address_id 这些关键值。

### 7. 一个好的自动化接口用例，怎样写才符合自动化标准？

## 接口测试补充题

### 1. 如果给你一个接口测试，你怎么写用例？

思路：

- 先看接口文档。
- 再从正常、异常、边界、权限、业务规则、数据校验设计。

回答：

> 我会先看接口文档，确认请求方式、URL、请求头、入参、返回字段和业务规则。然后从正常场景、必填参数、参数类型、边界值、权限鉴权、业务状态、数据落库几个角度设计用例。比如下单接口，除了正常下单，还要测库存不足、重复提交、地址无效、token 失效，以及下单成功后订单主表和订单明细表是否有数据。

### 2. 如果接口返回成功，但页面数据不对，你怎么排查？

思路：

- 先看前端请求参数。
- 再看接口响应。
- 再查数据库。
- 根据结果判断前端、后端或数据问题。

回答：

> 我会先判断问题出在接口、前端还是数据层。第一步看前端实际请求的接口和参数是否正确；第二步看接口响应数据是否正确；第三步查数据库确认底层数据。如果接口响应正确但页面展示不对，可能是前端字段映射、缓存或渲染逻辑问题；如果接口响应本身不对，就结合 SQL 查库和日志继续定位。

### 7. 字符串替换怎么做？

思路：

- 普通字符串替换可以用 `replace()`。
- 我的框架里更核心的是 YAML 占位符替换。
- 用 `re` 正则匹配 `${{变量名}}`。
- 通过 `match.group(1)` 取出变量名。
- 从 `ContextManager` 上下文里取真实值。
- 替换后再发送请求，用来解决 token、商品 ID、订单号等接口依赖。

回答：

> Python 普通字符串替换可以用 `replace()`。但在我的 `pytest_auto_api_my` 框架里，字符串替换主要用于接口参数依赖处理。比如 YAML 用例里写 `${{token}}`、`${{product_id}}`，框架在发送请求前会用 `re` 正则匹配这种占位符，通过 `match.group(1)` 取出变量名，再从 `ContextManager` 上下文里取真实 token、商品 ID、订单号进行替换。这样登录、商品、购物车、下单这些接口就能串成一条业务链路，不需要手动复制参数。

```python
PLACEHOLDER_PATTERN = re.compile(r"\$\{\{([^}]+)\}\}")

def repl(match):
    expr = match.group(1).strip()
    value = ContextManager.get(expr)
    return str(value)

result = PLACEHOLDER_PATTERN.sub(repl, "Bearer ${{token}}")
```

### 8. Python 如何处理异常？

思路：

- 用 `try...except`。
- 常见异常：请求超时、连接失败、响应不是 JSON、字段提取失败。
- 异常处理要保留请求和响应信息。

回答：

```python
try:
    result = response.json()
except ValueError:
    result = response.text
```

### 9. requests 怎么发送 GET 和 POST？

思路：

- `GET` 常用 `params`。
- `POST` JSON 请求体常用 `json`。
- 项目中统一封装 `send_request`，不每条用例直接写 requests。

回答：

> `requests.get()` 一般用于查询接口，参数常放 `params`；`requests.post()` 一般用于提交接口，JSON 请求体常放 `json`。在我的框架里，我没有在每条用例里直接写 `requests.get()` 或 `requests.post()`，而是封装了统一的 `send_request`，根据 YAML 里的 method、url、params、json 来发送请求。

```python
import requests

resp = requests.get("http://example.com/api", params={"page": 1})
resp = requests.post("http://example.com/login", json={"username": "test", "password": "123456"})
```

### e2 简述 web 工作原理以及 HTTP 协议工作的四个步骤

> Web 的工作原理：
>
> 1. 用户在浏览器输入 URL，浏览器解析地址
> 2. 浏览器通过 TCP/IP 与服务器进行连接，发送 HTTP 请求，请求由请求行、请求头、请求体组成
> 3. 服务器收到请求后，处理请求
> 4. 服务返回响应，响应包括状态行、响应头、响应体（JSON、HTML）
> 5. 浏览器进行解析与渲染图片

HTTP 工作的四个步骤：

> 1. 建立连接：客户端通过 TCP 与服务器建立连接（三次握手）
> 2. 发送请求：客户端发送 HTTP 请求（URL、方法、Header、Body）
> 3. 服务器处理请求并返回响应：服务器处理业务逻辑，返回状态码和数据
> 4. 关闭或保持连接：根据 Connection 设置关闭或复用连接

### e3 GET 请求和 POST 的区别

> GET 请求主要用作查询。
>
> POST 请求主要是用作表单提交。
>
> 参数的话，GET 主要放在请求 URL 上，POST 主要放在请求体上。

### e4 web 测试中，如何判断是前端的 bug 还是后端的 bug？

> 我会先看浏览器开发者工具里的请求参数和接口响应。
>
> 如果前端传给后端的参数就不对，比如字段缺失、格式错误、金额传错，那优先考虑前端问题。
>
> 如果请求参数正确，但接口返回的数据、业务码或数据库数据不对，那更偏后端或数据问题。
>
> 如果接口返回正确，但是页面展示错误，比如字段显示错、金额格式化错、按钮状态不对，那一般是前端渲染或字段映射问题。
>
> 核心业务还会结合数据库或后台数据一起确认，比如订单状态、支付状态、库存、金额是否一致。

### e5 h5 界面是怎么测试的？

> 功能上看按钮、链接、表单、提交、支付等流程是否正常。
>
> 兼容性上会关注不同手机系统、不同浏览器、微信内置浏览器、屏幕尺寸下是否显示正常。

>
> 异常方面会测弱网、断网、接口超时、重复点击、页面加载失败等情况。

### e6 拿到一个待检测的站，你觉得应该先做什么？

> 我会先了解这个网站的业务背景和核心流程，确认它是商城、内容站、后台系统还是活动页。
>
> 然后先做冒烟测试，确认首页、登录、核心入口、主要流程能不能正常打开和使用。
>
> 接着梳理测试范围和测试点，比如功能、页面展示、兼容性、接口交互、安全、性能和异常场景。
>
> 如果有需求文档或原型图，会先对照需求确认功能是否完整；如果没有文档，就先按用户路径和页面模块拆测试点。

### e7 请问你遇到过哪些印象深刻的 bug，接口测试出现 bug 的原因有哪些？

> 我遇到过比较典型的是订单金额不一致的问题。用户使用优惠券下单后，前端展示的实付金额和后台订单金额不一致。我先复现了用户操作路径，然后查看接口请求和响应，再到后台核对订单数据，发现是优惠券和会员价叠加规则前后端处理不一致。最后我提交 bug 时附上了复现步骤、截图、接口返回和后台数据，开发修复后我又回归了优惠券、会员价、退款等相关场景。
>
> 接口测试出现 bug 的常见原因有：
>
> - 参数校验不完整，比如必填项、类型、长度没有限制。
> - 业务规则处理错误，比如库存不足还能下单。
> - 权限校验不严，比如能查看别人的订单。
> - 数据状态判断错误，比如已取消订单还能支付。
> - 接口返回字段缺失或格式不符合文档。
> - 并发或重复提交导致重复下单、重复扣库存。
> - 数据库写入失败或数据不一致。

### e8 你在做项目中有做过压力测试吗，怎么做？

> 有做过基础的接口压力测试，主要是用 JMeter 对核心接口进行并发测试，比如登录、商品查询、下单接口。
>
> 我的思路是先准备测试环境和测试数据，比如账号、token、商品 ID、地址 ID。然后在 JMeter 里配置线程组、HTTP 请求、请求头和请求参数，通过不同并发数模拟用户请求。
>
> 执行过程中重点关注响应时间、TPS、吞吐量、错误率，同时观察服务器 CPU、内存、数据库响应是否异常。
>
> 如果是下单接口，还会关注是否出现重复下单、库存超卖、订单状态异常、接口超时等问题。

### e9 如何对一个页面进行测试？

> 我会从功能、UI、兼容性、易用性、接口交互、异常场景、安全几个方面测。
>
> 功能上看页面按钮、链接、查询、提交、重置等功能是否正常。
>
> UI 上看布局、文字、图片、颜色、对齐、提示信息是否符合原型。
>
> 兼容性上看不同浏览器、不同分辨率、移动端适配是否正常。
>
> 易用性上看操作是否符合用户习惯，错误提示是否清楚。
>
> 接口交互上看请求参数、响应数据和页面展示是否一致。
>
> 异常场景会测空数据、网络异常、重复点击、接口失败、加载超时。
>
> 安全方面会关注权限控制、敏感信息展示、SQL 注入、XSS 等基础问题。

### e10 请简述一下手动测试与自动化测试的优缺点

> 手动测试的优点是灵活，适合新功能、需求变化频繁、UI 体验、探索性测试；缺点是重复执行效率低，容易受人工状态影响。
>
> 自动化测试的优点是执行快、可重复、适合回归测试，可以提升稳定功能的测试效率；缺点是前期脚本编写和维护成本较高，不适合需求频繁变化或页面经常改动的功能。
>
> 我的理解是，手工测试适合发现问题和验证新功能，自动化测试适合稳定业务的回归，两者是互补关系。

### e11 怎么做 WEB 表单测试，从流程，数据，标准等去分析？

> Web 表单测试我会从表单流程、字段数据、校验规则、提交结果和异常安全几个方面测。
>
> 流程上看进入页面、填写表单、提交、保存、取消、返回、重复提交是否正常。
>
> 数据上看必填项、非必填项、默认值、空值、超长字符、特殊字符、中文英文、数字、日期、手机号、邮箱等格式。
>
> 标准上看错误提示是否清楚，提示位置是否正确，字段长度和格式是否符合需求。
>
> 提交后要看页面提示、接口响应、数据库落库是否一致。
>
> 异常方面会测网络中断、接口失败、重复点击提交、无权限提交。
>
> 安全方面会测 SQL 注入字符、XSS 脚本、敏感字段是否明文展示。

### e12 怎么做 SQL 安全测试，特别是 SQL 注入的测试？

> SQL 注入测试主要看系统是否把用户输入直接拼接到 SQL 里。
>
> 我会在登录框、搜索框、URL 参数、接口参数里输入一些特殊字符或 SQL 片段，比如：
>
> ```sql
> ' or '1'='1
> admin' --
> ' union select 1,2,3 --
> ```
>
> 然后观察系统是否报数据库错误、是否绕过登录、是否返回异常数据。
>
> 正常系统应该对输入做校验、转义或使用预编译参数，不能直接拼接 SQL。
>
> 测试时要注意只在测试环境操作，不能破坏真实数据。

### e13 请针对 https://yyw.ai 进行测试，可以选择其中一个栏目进行深入的测试，要求深入并尽量全面。

> 如果让我测试这个网站，我会先选择“登录 / 注册”或“核心业务入口”作为重点模块，因为这是用户进入系统的关键链路。
>
> 以登录注册为例，我会从以下几个方面测试：
>
> 1. 功能流程：正常注册、正常登录、退出登录、忘记密码、验证码发送和校验。
> 2. 输入校验：手机号、邮箱、密码为空，格式错误，密码长度边界，特殊字符。
> 3. 验证码：验证码是否过期，错误验证码是否拦截，重复发送是否有限制。
> 4. 异常场景：账号不存在、密码错误、账号被禁用、网络异常、接口超时。
> 5. 安全测试：多次输错密码是否限制，是否防暴力破解，密码是否明文传输，是否存在 SQL 注入和 XSS。
> 6. 兼容性：Chrome、Edge、移动端浏览器和不同分辨率下是否正常。
> 7. UI 和易用性：错误提示是否清楚，按钮状态是否正确，加载中是否有提示。
> 8. 接口和数据：请求参数是否正确，接口返回是否符合预期，登录成功后 token 或 session 是否正常。
>
> 如果测试网站首页或某个栏目，我会补充栏目导航、内容展示、搜索筛选、详情页跳转、图片加载、分页、空数据、接口异常和页面加载性能等测试点。

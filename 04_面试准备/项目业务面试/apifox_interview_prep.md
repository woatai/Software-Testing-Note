# Apifox 面试速背稿

这份笔记的目标不是背工具菜单，而是把 Apifox 说成你做接口测试、接口关联、断言校验和业务链路验证的工具。

---

## 一、Apifox 是什么

可以直接背：

> Apifox 是一个集接口文档、接口调试、接口测试、自动化用例和接口管理于一体的工具。我平时主要用它做接口联调、环境变量管理、前后置脚本、变量提取、接口断言，以及业务链路的串联验证。

更短版：

> 我一般把 Apifox 当作接口调试和接口测试工具来用，重点会用到环境变量、前后置脚本、变量提取、断言和接口链路串联。

---

## 二、面试官最常问的 Apifox 问题

### 1. 你平时怎么用 Apifox

可以这样答：

> 我平时会先在 Apifox 里配置测试环境，比如 `base_url`、账号、密码、token、商品 ID、订单号这些变量；然后先跑登录接口，通过后置脚本或者提取变量把 token 保存下来，后面的接口直接引用 token。对于关键业务接口，我会加断言校验返回状态、关键字段和业务结果。如果是完整链路，比如登录、商品、下单、订单查询，我会通过变量传递把这些接口串起来。

### 2. Apifox 环境变量、全局变量、临时变量有什么区别

可以这样答：

> 环境变量一般和不同环境相关，比如测试环境、预发环境、生产环境的 `base_url`、token、账号信息；全局变量是多个项目或多个环境都可能共用的变量；临时变量通常只在当前调试或当前请求里临时使用，不适合长期保存。实际工作里我用得最多的是环境变量，因为接口测试通常要区分测试环境和预发环境。

### 3. 你怎么处理 token

可以这样答：

> 一般是先调用登录接口，在登录接口执行成功后，通过后置脚本或者变量提取把返回里的 token 存成环境变量。后续请求统一在请求头里引用，比如 `Authorization: Bearer {{token}}`。这样做的好处是 token 变化后不用每个接口手动改。

### 4. 你怎么做接口关联

可以这样答：

> 我通常会把上一个接口返回的关键字段提取成变量，再给下一个接口使用。比如登录后提取 token，下单后提取订单号 `order_no`，再在订单查询接口里引用。这样就能把一整条业务链路串起来，而不是单独调每个接口。

### 5. 你怎么做接口断言

可以这样答：

> 我做断言一般会看三层。第一层是接口是否请求成功，比如状态码、业务码是否正确；第二层是返回结构和关键字段是否存在；第三层是业务结果是否符合预期，比如返回的订单号是否存在、查询结果里是否包含刚创建的订单、金额字段是否正确。

### 6. 你会用前置脚本和后置脚本做什么

可以这样答：

> 前置脚本我常用来处理请求头、公共参数、签名或者 token 注入；后置脚本主要用来提取变量、打印调试信息和写断言。比如登录成功后，我会在后置脚本里把 token 提取成环境变量；下单后会提取订单号给下一个接口继续使用。

### 7. 你会不会用 Apifox 做数据库校验

可以这样答：

> 现实工作里，Apifox 主要还是用来跑接口链路、做变量提取和接口断言。数据库校验很多时候我会结合 Navicat、DataGrip 或命令行去查库确认结果。Apifox 也可以接数据库做校验，但我更常见的是用它跑接口，再用 SQL 抽样验证订单、余额、优惠券、会员权益这些核心数据是否真实落库。

### 8. 你怎么用 Apifox 跑一条完整业务链路

可以这样答：

> 我的做法一般是先梳理链路里的关键接口和关键变量。比如登录拿 token，商品详情拿商品 ID，下单拿订单号，订单查询再用订单号做校验。然后在 Apifox 里把这些接口按顺序组织起来，通过环境变量和后置提取做参数传递，最后在关键节点加断言，必要时再结合 SQL 去校验数据库结果。

## 三、你这次项目里可以直接说的 Apifox 实战

结合这次 CRMEB 练习，可以这样答：

> 我最近练习过一条比较完整的接口测试链路。先通过登录接口获取 token，在后置脚本里提取 `data.token` 保存成环境变量；然后调用商品列表和商品详情接口验证商品数据；接着走购物车、订单确认、订单金额计算和创建订单接口，在下单成功后提取 `orderId` 作为 `order_no`；最后再调用订单详情接口做结果校验。对于关键接口，我会加状态码、关键字段和业务结果断言，链路跑完后再用 SQL 查订单主表和订单明细表确认是否真实落库。

## 四、面试里最常见的 Apifox 场景题

### 1. 登录成功后，后续接口需要带 token，你怎么处理

标准回答：

> 我会先调登录接口，然后在后置脚本里从返回 JSON 里提取 token，保存成环境变量。后面的接口统一在请求头里引用这个变量，比如 `Authorization: Bearer {{token}}`。这样 token 变化后不需要每个接口手动更新。

### 2. 下单成功后，订单查询接口要用到订单号，你怎么处理

标准回答：

> 我会在创建订单接口的后置脚本里提取返回中的订单号，比如 `orderId`，保存成环境变量 `order_no`。然后在订单详情或者订单查询接口里直接引用 `{{order_no}}`，这样就完成了接口之间的参数关联。

### 3. 你怎么判断一个接口测试通过了

标准回答：

> 我不会只看 HTTP 200。一般会同时看业务状态码、返回结构、关键字段和业务结果。比如下单接口除了返回成功，还要确认是否返回了订单号；订单查询接口除了返回成功，还要确认查询结果里是否包含刚创建的订单。如果是金额、余额、优惠券这种核心场景，我还会结合数据库校验。

### 4. 如果接口依赖比较多，你怎么组织测试

标准回答：

> 我会先找出这条链路里的核心依赖和关键变量，比如 token、商品 ID、地址 ID、订单号。然后把前置依赖接口先跑通，再通过环境变量和脚本把参数串起来。这样做的重点是先解决依赖关系，再做断言，而不是每个接口孤立地去测。

### 5. 如果后端接口返回成功，但页面数据不对，你怎么排查

标准回答：

> 我会先确认接口请求参数是否正确、接口返回数据是否正确；如果接口返回本身就不对，就优先看后端逻辑或数据问题；如果接口返回对，但页面展示不对，再看前端渲染、字段映射或者缓存问题。必要时我会用 SQL 查底层数据，确认是接口问题、前端问题，还是数据本身就不对。

## 五、Apifox 里你要会说的几个功能点

### 1. 环境管理

- 管理不同环境的 `base_url`
- 管理测试账号、密码、token
- 管理商品 ID、订单号这类链路变量

### 2. 变量提取

- 从登录响应里提取 token
- 从下单响应里提取订单号
- 从商品详情里提取商品 ID 或规格 ID

### 3. 前置脚本

- 自动注入 token
- 处理公共请求头
- 处理时间戳、签名、随机数

### 4. 后置脚本

- 提取返回字段
- 写断言
- 调试打印返回结果

### 5. 断言

- 状态码断言
- 业务码断言
- 关键字段断言
- 返回值包含断言

## 六、你这次可以直接背的脚本思路

### 1. 登录后提取 token

思路：

> 登录接口成功后，在后置脚本里从返回 JSON 里取 `data.token`，写入环境变量 `token`。

### 2. 后续接口自动带 token

思路：

> 在前置脚本里读取环境变量 `token`，自动拼到请求头 `Authorization: Bearer {{token}}`。

### 3. 下单后提取订单号

思路：

> 在创建订单接口成功后，把返回中的 `orderId` 提取为环境变量 `order_no`，供订单详情接口继续使用。

### 4. 订单查询接口做断言

思路：

> 我会断言接口是否请求成功，再断言查询结果里是否包含当前 `order_no`，这样能确认这次创建的订单确实能被查到。

## 七、Apifox 常见易错点

- 只看 HTTP 状态码，不看业务码
- 登录后没有正确提取 token
- token 放错位置，比如该放请求头却放到了参数里
- 接口关联时变量名写错
- 断言只断言成功，不断言关键业务字段
- 测试链路跑通了，但没有做数据库校验
- 不区分环境变量和全局变量，导致值串环境

## 八、面试里怎么回答“你怎么用 Apifox 做接口测试”

可以直接背：

> 我用 Apifox 做接口测试时，一般会先配置测试环境，把 `base_url`、账号、密码、token、商品 ID、订单号这些变量管理起来。然后先跑登录接口，通过后置脚本提取 token 保存成环境变量，后续接口统一引用。对于关键业务场景，我会通过前后置脚本做接口关联，比如下单后提取订单号给订单查询接口使用。断言方面，我不会只看 HTTP 状态码，还会校验业务码、关键字段和业务结果。对于订单、金额、会员权益、优惠券核销这类核心场景，我还会结合 SQL 查库，确认数据是否真实落库、状态是否正确、金额是否一致。

## 九、面试里怎么回答“Apifox 和 Postman 有什么区别”

可以这样答：

> 我两个都接触过。Postman 更偏经典接口调试工具，Apifox 除了调试接口，还把接口文档、调试、测试、用例管理和自动化整合到了一起，团队协作会更方便一些。对我来说，如果是个人快速调试接口，两个都能用；如果是结合文档、环境、变量管理和链路测试，Apifox 会更顺手。

这个问题不用讲得特别深，重点是表达你会用，不要硬踩另一个工具。

## 十、面试前最后提醒

### 1. 不要把 Apifox 说成“只是发请求的工具”

尽量说：

> 我会用 Apifox 管理环境变量、做接口关联、写断言、跑业务链路，并结合 SQL 做结果校验。

### 2. 不要只说“我会调接口”

尽量说：

> 我会把登录、下单、查询这类接口串成完整链路，验证返回结果和数据库结果是否一致。

### 3. 回答时尽量带业务词

比如：

- token 提取
- 订单号关联
- 金额校验
- 优惠券核销
- 会员权益
- 数据落库

这样面试官会觉得你不是只会点工具，而是真的在用它做业务验证。

# 云起科技 CRM 接口测试面试准备

## 项目介绍话术

这个项目是我基于 `pytest + requests + yaml + allure` 搭建的接口自动化框架，主要验证一个下单主流程。框架的核心链路是：从 YAML 读取接口用例，解析上下文占位符，统一发送 HTTP 请求，执行状态码和 JSONPath 业务断言，再把响应里的 token、商品信息、购物车和订单相关字段提取到上下文，供后续接口复用。

它比较适合回归测试和主流程稳定性验证。比如 CRM 会员运营系统里常见的登录、会员查询、优惠券发放、短信触达、标签更新、订单同步、活动效果查询，都可以按类似方式维护成 YAML 用例，并通过上下文串起完整业务流。

## 和云起科技岗位的匹配点

- `ApiFox / Postman`：可以先用工具调通接口、确认参数和响应，再沉淀到 YAML 自动化用例。
- `接口功能测试`：框架支持状态码断言、JSONPath 字段断言、响应字段提取和多接口流程串联。
- `接口稳定性`：请求返回结构里记录了请求调试信息和接口耗时，失败时能快速定位请求参数、URL、响应体。
- `自动化测试框架`：已具备用例数据分离、统一请求封装、统一断言、统一提取、上下文管理、Allure 报告。
- `CRM 业务理解`：可以把会员、优惠券、标签、短信、活动、订单数据这些接口按业务链路设计用例。
- `SQL 能力`：面试时可以补充说明，接口断言之外会通过 MySQL 校验会员标签、优惠券库存、发放记录、短信发送记录、订单同步状态等落库数据。

## 高频面试题和回答要点

### 1. 你这个接口自动化框架怎么设计的？

回答要点：

我的框架按职责拆了几层。YAML 负责维护接口数据；`case_runner` 是单接口执行入口；`RequestControl` 统一处理 host 拼接、headers、token 注入、请求发送和调试信息；`AssertControl` 统一做状态码和 JSONPath 断言；`ExtractControl` 把响应里的字段提取到 `ContextManager`，用于后续接口。这样测试代码只需要准备 YAML 和调用 `run_case`，不会在每个测试里重复写请求、断言和提取逻辑。

### 2. 为什么要做 YAML 数据驱动？

回答要点：

接口测试里参数、断言和提取规则变化比较频繁，把它们放在 YAML 里，测试代码会更稳定。比如 CRM 系统里新增一个会员标签接口，我只需要新增 YAML case，再通过 `run_case` 执行。公共字段可以放在 `case_common`，减少重复配置。

### 3. 多接口依赖怎么处理？

回答要点：

我用 `ContextManager` 做流程上下文。比如登录接口提取 `token`，商品列表提取 `product_id` 和 `unique`，后面的购物车、订单确认接口通过 `${{token}}`、`${{product_id}}` 这类占位符引用。每条流程开始和结束都会通过 fixture 清空上下文，避免用例之间互相污染。

### 4. 如果接口失败，你怎么定位？

回答要点：

框架会在返回结构里保存 `request_debug`，包括 method、url、headers、params、data、json。断言失败时错误信息会打印请求和响应。现在也会把请求、响应、耗时写入 Allure 附件，所以能看到具体是哪一步、哪个接口、哪个字段和预期不一致。

### 5. CRM 会员系统你会重点测哪些接口？

回答要点：

我会按业务域拆：会员注册/登录、会员资料、会员等级、积分、优惠券、标签、人群分层、短信触达、活动配置、订单同步、数据看板。重点关注会员身份是否唯一、标签是否准确、优惠券库存和领取限制、短信发送状态、订单数据同步一致性，以及活动后的转化数据统计是否正确。

### 6. 优惠券发放接口怎么设计测试点？

回答要点：

功能层面测正常领取、重复领取、库存不足、过期券、未开始券、会员等级限制、指定人群限制。数据层面查优惠券库存是否扣减、用户领券记录是否生成、状态是否正确。并发或性能层面关注多人同时领取时是否超发。

### 7. 你怎么做接口性能测试？

回答要点：

我会先用 Postman 或 ApiFox 做单接口调试，再用 JMeter 做并发压测。关注平均响应时间、P95/P99、吞吐量、错误率、数据库慢 SQL、CPU 和内存。比如营销活动开始时，优惠券领取、会员查询、短信发送记录接口可能会有流量峰值，需要设置阶梯压测和稳定性压测。

### 8. SQL 在测试里怎么用？

回答要点：

接口返回只代表一部分结果，关键业务还要查库验证。比如领取优惠券后，查用户券表是否新增记录、券库存是否减少；会员打标签后，查标签关系表；订单同步后，查订单主表和明细表状态。SQL 也可以用于准备测试数据和清理测试数据。

### 9. 你这个框架还有什么不足？

回答要点：

目前它适合接口主流程回归，但还可以继续增强：增加数据库断言、环境配置校验、失败重试策略、接口性能基线、并发测试脚本、测试数据自动清理，以及把 ApiFox 导出的接口定义转换成 YAML 用例。这个回答要真实，不要说已经全部完成。

### 10. 如果需求不清楚，你怎么处理？

回答要点：

我会先从业务规则、接口文档、原型和历史数据确认边界。比如会员标签规则，要问清楚标签来源、更新时机、覆盖还是追加、失败是否重试、是否影响后续营销人群。确认后再写测试点和用例，避免只按接口字段测，而漏掉业务约束。

## 可以主动展示的项目亮点

- YAML 数据驱动，测试数据和测试代码分离。
- 上下文机制支持多接口业务流串联。
- 统一断言引擎，支持状态码和 JSONPath 业务字段校验。
- 统一提取引擎，支持普通 JSONPath 和按条件查找目标字段。
- 请求封装里支持默认 host、默认 headers、token 注入和调试信息。
- Allure 报告里能查看接口请求、响应和耗时。

## 需要避免的说法

- 不要说这个项目已经覆盖 CRM 全业务，它目前主要是电商下单主流程。
- 不要说已经实现完整性能测试，目前只是接口耗时记录，JMeter 性能压测可以作为下一步能力说明。
- 不要说环境切换、数据库断言、数据清理都很完善，除非后续继续补齐。
- 不要只讲工具名，要把工具和业务场景连起来，比如优惠券、会员标签、短信触达、订单同步。

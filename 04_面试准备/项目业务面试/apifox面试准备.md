# Apifox 接口测试面试准备

## 这份笔记怎么用

这份是 5-7k 接口测试岗位的面试前冲刺版，重点不是讲高级功能，而是把 Apifox 的基础能力说清楚：

- 会调接口。
- 会配环境。
- 会传 token。
- 会做接口关联。
- 会写基础断言。
- 会结合业务链路和 SQL 校验结果。

一句话介绍：

> Apifox 是一个集接口文档、接口调试、接口测试、环境变量、前后置脚本和断言于一体的接口测试工具。我主要用它做接口调试、接口联调、接口关联、断言校验和业务链路验证。

## 我的 Apifox 使用流程

可以按这个顺序讲：

1. 先看接口文档，确认请求方式、URL、请求头、入参和返回字段。
2. 配置环境变量，比如 `base_url`、账号、密码、token。
3. 先调通单接口，确认参数和响应结构正确。
4. 登录接口提取 token，后续接口统一引用。
5. 做接口关联，比如下单后提取订单号给订单查询接口使用。
6. 给关键接口加断言，不只看 HTTP 200。
7. 核心业务再结合 SQL 查库，确认是否真实落库。

## 高频面试题

### 1. 你平时怎么用 Apifox？

> 我平时会先在 Apifox 里配置测试环境，比如 `base_url`、账号、密码、token、商品 ID、订单号这些变量。然后先跑登录接口，通过后置脚本或变量提取把 token 保存下来，后面的接口直接引用 token。对于关键业务接口，我会加断言校验状态码、业务码、关键字段和业务结果。如果是完整链路，比如登录、商品、购物车、下单、订单查询，我会通过变量传递把这些接口串起来。

### 2. 拿到一个接口，你怎么在 Apifox 里测试？

> 我会先看接口文档，确认请求方式是 GET 还是 POST，URL 是什么，需要哪些 headers，请求参数放在 query、body 还是路径里。然后在 Apifox 里填写请求参数并发送请求，先确认接口能不能调通，再看返回状态码、业务码、message、data 这些字段是否符合预期。最后我会补充正常场景、异常参数、必填校验、权限校验和业务规则校验。

### 3. GET 和 POST 有什么区别？

> GET 一般用于查询数据，参数通常放在 URL 的 query 里；POST 一般用于新增、提交或复杂操作，参数通常放在 body 里。接口测试时，我不会只按请求方式判断，还会结合接口文档确认参数位置、Content-Type 和业务含义。

### 4. 接口参数一般放在哪些位置？

> 常见位置有四种：query 参数、path 路径参数、body 请求体、headers 请求头。比如分页参数常放 query，用户 ID 可能放 path，登录账号密码通常放 body，token 一般放 headers。

### 5. 常见的 Content-Type 有哪些？

> 接口测试里常见的是 `application/json` 和 `application/x-www-form-urlencoded`。如果是 JSON 请求，请求体一般是 JSON 格式；如果是表单提交，就是 key-value 格式。Apifox 里要根据接口文档选择正确的 body 类型，不然接口可能会解析不到参数。

### 6. 环境变量、全局变量、临时变量有什么区别？

> 环境变量通常和不同环境有关，比如测试环境、预发环境的 `base_url`、token、账号信息；全局变量是多个接口或多个项目都可能共用的值；临时变量一般只在当前请求或当前调试过程中临时使用。实际测试里我用得最多的是环境变量，因为接口测试经常要区分不同环境。

### 7. 登录成功后，后续接口需要 token，你怎么处理？

> 我会先调用登录接口，在登录接口成功后，通过后置脚本或变量提取把返回里的 token 保存成环境变量。后续接口统一在请求头里引用，比如 `Authorization: Bearer {{token}}`，或者按项目要求放 `Authori-zation`、`token` 这类 header。这样 token 变化后不用每个接口手动修改。

### 8. 什么是接口关联？Apifox 里怎么做？

> 接口关联就是上一个接口返回的数据，要给下一个接口继续使用。比如登录接口返回 token，下单接口返回订单号，后面的订单查询接口要用这个订单号。在 Apifox 里，我会通过后置脚本或提取变量，把返回字段保存成环境变量，再在后续接口里用 `{{变量名}}` 引用。

### 9. 下单成功后，订单查询接口要用订单号，你怎么处理？

> 我会在创建订单接口的后置处理中提取返回的订单号，比如 `order_id` 或 `order_no`，保存成环境变量。然后订单详情或订单查询接口直接引用这个变量。这样可以验证“创建订单”和“查询订单”这两个接口是不是能串起来。

### 10. 你怎么做接口断言？

> 我做断言一般分三层。第一层是 HTTP 状态码，比如是不是 200；第二层是业务码和返回结构，比如 code、message、data 是否符合预期；第三层是业务结果，比如登录是否返回 token，下单是否返回订单号，查询结果里是否包含刚创建的订单。不能只看 HTTP 200，因为 HTTP 成功不代表业务一定成功。

### 11. 你怎么判断一个接口测试通过？

> 我会看请求是否成功、业务码是否正确、返回结构是否符合接口文档、关键字段是否存在、业务结果是否符合预期。比如下单接口除了返回成功，还要有订单号；订单查询接口除了返回成功，还要能查到刚创建的订单。核心业务还要结合 SQL 查库，确认数据是否真实落库。

### 12. 前置脚本和后置脚本分别用来做什么？

> 前置脚本一般在请求发送前执行，可以用来处理公共请求头、时间戳、签名、随机参数、token 注入。后置脚本一般在接口响应后执行，常用来提取变量、写断言、打印调试信息。比如登录接口成功后，我会在后置脚本里提取 token。

### 13. 如果接口依赖比较多，你怎么组织测试？

> 我会先梳理业务链路和关键变量，比如 token、商品 ID、地址 ID、订单号。然后先跑通前置依赖接口，再通过变量提取和环境变量把参数串起来。重点是先解决接口依赖关系，再做断言，而不是每个接口孤立地测。

### 14. 如果接口返回成功，但页面数据不对，你怎么排查？

> 我会先确认前端请求的接口和参数是否正确，再看接口响应数据是否正确。如果接口返回本身不对，优先排查后端逻辑或数据问题；如果接口返回正确但页面展示不对，再看前端字段映射、渲染逻辑或缓存问题。必要时我会用 SQL 查底层数据，确认是接口问题、前端问题，还是数据本身不对。

### 15. Apifox 和 Postman 有什么区别？

> 两个工具都可以做接口调试。Postman 更偏经典接口调试工具，Apifox 除了调试接口，还集成了接口文档、环境管理、接口测试、用例管理和团队协作。对我来说，个人快速调接口两个都能用；如果要结合接口文档、变量管理、链路测试和团队协作，Apifox 会更方便。

## 接口用例设计

如果面试官问“给你一个接口，你怎么写用例”，可以这样答：

> 我会从正常场景、必填参数、参数类型、边界值、权限鉴权、业务规则、数据校验几个角度设计。

- 正常场景：参数正确，接口返回成功。
- 必填参数：缺少必填字段是否提示。
- 参数类型：数字传字符串、日期格式错误、枚举值错误。
- 边界值：数量、金额、长度、分页参数。
- 权限鉴权：不传 token、token 过期、越权访问。
- 业务规则：库存不足、重复提交、状态不允许操作。
- 数据校验：接口返回、页面展示、数据库结果是否一致。

## 项目实战说法

结合 CRMEB 或商城接口测试，可以这样说：

> 我练习过一条商城接口链路。先通过登录接口获取 token，再调用商品列表和商品详情接口验证商品数据；接着走购物车、订单确认、订单金额计算和创建订单接口；下单成功后提取订单号，最后调用订单详情接口做结果校验。关键接口会加状态码、业务码、关键字段和业务结果断言，链路跑完后再用 SQL 查订单主表和订单明细表，确认是否真实落库。

## 常见易错点

- 只看 HTTP 状态码，不看业务码。
- token 没提取成功，后续接口全部失败。
- token 放错位置，比如该放 headers 却放到 body。
- 环境变量和全局变量混用，导致串环境。
- 接口关联时变量名写错。
- 断言只断言成功，不断言关键业务字段。
- 链路跑通了，但没有做数据库校验。

## 最后速背

> 我用 Apifox 做接口测试时，一般会先配置环境变量，比如 `base_url`、账号、密码、token。然后先调通单接口，确认请求方式、参数位置、headers 和响应结构。登录成功后，我会提取 token 给后续接口使用；如果是下单链路，会继续提取商品 ID、地址 ID、订单号这些变量，把接口串起来。断言方面，我不会只看 HTTP 200，还会看业务码、关键字段和业务结果。对于订单、金额、库存这类核心场景，我还会结合 SQL 查库，确认数据是否真实落库、状态是否正确。

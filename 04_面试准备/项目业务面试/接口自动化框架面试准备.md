# 接口自动化框架面试准备

## 这份笔记怎么用

这份是面试前冲刺版，只保留最容易被问、最适合结合我自己的框架 `E:\pytest_auto_api_my` 回答的问题。

项目定位要说准：

> 我自己基于 `pytest + requests + yaml` 搭建了一个接口自动化框架，用 CRMEB 商城接口做练习场景，主要覆盖登录、商品、购物车、地址、下单这些主流程。这个项目重点不是覆盖所有业务，而是验证我对接口自动化框架分层、数据驱动、接口关联、断言和失败定位的理解。

## 回答主线

遇到接口自动化框架题，不要散着背工具名，按这个顺序讲：

1. 目标：减少重复手工调接口，沉淀稳定回归用例。
2. 技术栈：`pytest + requests + yaml`。
3. 分层：YAML 用例、pytest 执行、请求封装、断言、提取、上下文、配置。
4. 链路：登录提取 token，商品提取商品 ID，购物车和下单接口引用上下文变量。
5. 定位：看请求参数、响应体、断言失败点、上下文变量，必要时结合 SQL 查库。
6. 不足：数据库断言、数据清理、失败重试、报告展示还可以继续增强。

## 我的框架怎么讲

可以直接说：

> 我的框架按职责分层。`data/` 放 YAML 用例数据，`test_case/` 放 pytest 测试入口，`util/requestsUtils/` 做 requests 请求封装，`util/assertion/` 做断言，`util/extract/` 做响应字段提取，`util/context/` 管理上下文，`common/` 放环境和路径配置。这样接口数据和测试代码分离，后续新增接口时主要维护 YAML，不需要每个测试脚本里重复写请求、断言和提取逻辑。

执行流程：

```text
读取 YAML 用例
    ↓
合并公共参数
    ↓
替换上下文变量
    ↓
统一发送 HTTP 请求
    ↓
执行状态码 / JSONPath 断言
    ↓
提取响应字段到上下文
    ↓
后续接口继续引用
```

## 高频面试题

### 1. 用 Python 如何搭建接口自动化框架？

> 我会先明确框架目标：减少重复手工调接口、沉淀稳定回归用例、支持多接口业务流串联，并且失败后方便定位。技术选型上，我会用 `pytest + requests + yaml`。`pytest` 负责用例组织、参数化和 fixture；`requests` 负责发送 HTTP 请求；`yaml` 负责维护接口路径、请求参数、断言和提取规则。

结合我的项目：

> 我的 `E:\pytest_auto_api_my` 就是按这个思路搭的。`data/` 目录放登录、商品、购物车、订单这些 YAML 用例，`test_case/` 负责 pytest 执行入口，`util/requestsUtils/` 做请求封装，`util/assertion/` 做断言，`util/extract/` 做字段提取，`util/context/` 管理上下文。这样回答时能落到真实目录，不像只是在背技术栈。

### 2. 自动化搭建的原理是什么？接口自动化是怎么实现的？

> 接口自动化的原理，就是把手工调接口的动作标准化、数据化、可重复执行。手工测试时，我们会准备请求参数、发送请求、查看响应、提取下一个接口需要的数据、判断结果是否正确；自动化框架就是把这些动作拆成模块，由程序根据 YAML 用例自动执行。

结合我的项目：

> 比如 `order_main_flow.yaml` 这种流程用例，前面的登录接口先提取 token，商品接口提取商品 ID，后面的购物车、地址、下单接口再引用这些上下文变量。这样不是每个接口孤立执行，而是按真实业务流程串起来。

### 3. 是自己写框架，还是基于公司已有框架写脚本？

> 这个项目是我自己基于 `pytest + requests + yaml` 搭建的练习框架，不是公司已有框架。自己搭的好处是，我能更清楚理解接口自动化框架的底层流程，比如用例怎么读取、请求怎么封装、断言怎么设计、接口依赖怎么传递、上下文怎么清理。

更稳妥的补充：

> 如果工作中公司已有成熟框架，我会优先遵循公司规范，在现有框架里补充业务脚本、断言和数据维护，不会重复造轮子。自己搭框架是为了掌握原理，工作中更重要的是用合适的方式提高回归效率和维护效率。

### 4. 多接口依赖怎么处理？

> 我用上下文管理器处理接口依赖。比如登录接口返回 token 后，把 token 提取到上下文；商品接口返回商品 ID 后，也提取到上下文。后面的购物车、确认订单、创建订单接口，就可以通过占位符引用这些值。这样一条业务链路可以按接口执行顺序自动传递数据。

结合我的项目：

> 下单主流程里最典型的依赖就是 token、product_id、cart_id、address_id、order_id。如果这些值手动复制，用例会很难维护，所以我会让上游接口提取，下游接口通过占位符引用。

### 5. 断言怎么设计？

> 我一般分三层断言。第一层是 HTTP 状态码，比如是不是 200；第二层是业务码和返回结构，比如 code、message、data 是否符合预期；第三层是关键业务字段，比如 token 是否存在、订单号是否返回、商品数量或金额是否符合预期。只断言 HTTP 200 不够，因为很多接口 HTTP 成功但业务可能失败。

结合我的项目：

> 登录接口不能只看请求成功，还要判断 token 是否返回；下单接口不能只看返回成功，还要判断订单号是否返回。后续如果增强数据库断言，还可以查订单主表和订单明细表，确认真的落库。

### 6. 写自动化过程中遇到的问题及挑战，如何解决？

> 我遇到比较典型的问题主要有三类：接口依赖、测试数据不稳定、失败定位不清楚。

- 接口依赖：用上下文管理器解决，把 token、商品 ID、订单号这些值提取出来给后续接口使用。
- 数据不稳定：尽量选择稳定测试数据，把关键数据放到 YAML 或配置里管理，后续可以增强数据初始化和清理。
- 失败定位：请求封装里保留 method、url、headers、params、body、响应体等信息，失败时先看请求是否正确，再看响应业务码和断言字段。

结合我的项目：

> 我在做下单主流程时，最容易出问题的是上游变量没有提取到，导致后面的购物车或订单接口参数为空。所以定位时我会重点检查上下文里有没有 token、product_id、cart_id、address_id 这些关键值。

### 7. 一个好的自动化接口用例，怎样写才符合自动化标准？

> 我认为一个好的接口自动化用例，不能只是能跑通，还要稳定、独立、可维护、断言有效、失败好定位。

可以从这几方面说：

- 用例目标清楚，一条用例验证一个明确场景。
- 前置条件稳定，不依赖临时手工数据。
- 请求参数可维护，公共参数不要重复写。
- 断言有效，不只断言 HTTP 200，还要校验业务码和关键字段。
- 有接口依赖时，通过提取和上下文传递，不手动复制 token、订单号。
- 失败信息清楚，能看到请求参数、响应体和断言失败原因。

结合我的项目：

> 在我的框架里，好的用例不是把请求参数硬编码在测试函数里，而是写在 YAML 中，测试函数只负责读取和执行。这样如果接口路径、商品 ID、断言字段变化，优先改 YAML，不用频繁改 Python 测试代码。

### 8. 自动化脚本的业务占比、测试占比是多少？

> 我不会盲目追求自动化覆盖率 100%。接口自动化更适合覆盖稳定、高频、核心、回归价值高的接口和业务链路。

结合我的项目可以这样答：

> 我当前这个框架重点覆盖商城接口主流程，所以更偏核心业务链路，比如登录、商品、购物车、地址、下单。可以理解为 70% 放在核心业务流程，30% 放在单接口基础校验和异常场景。真正工作中比例要看项目阶段，如果接口还在频繁变化，自动化比例不能太高；如果接口比较稳定，就可以逐步提高自动化回归覆盖率。

加分点：

> 自动化脚本的价值不在数量，而在是否覆盖核心风险、是否稳定、是否能减少回归成本。

### 9. Apifox 调通接口后，怎么沉淀到自动化用例？

> 我会先用 Apifox 做接口调试和链路验证，确认请求参数、响应结构、变量关联和断言思路都没问题。等接口稳定后，再把它沉淀成 YAML 自动化用例，放到 pytest 框架里做回归。

结合我的项目：

> 比如先在 Apifox 里调通登录、商品、购物车、下单接口，确认 token、商品 ID、订单号这些字段怎么取；然后在 YAML 里写对应的请求参数、断言和提取规则。这样 Apifox 负责前期调试，pytest 框架负责后期自动化回归。

### 10. 这个框架还有什么不足，后续怎么优化？

> 目前框架已经具备用例数据分离、请求封装、统一断言、字段提取和上下文传递，适合做主流程接口回归。但它还可以继续增强：数据库断言、测试数据自动清理、失败重试、日志和报告展示、多环境配置校验，以及把 Apifox 调通的接口更方便地转换成 YAML 用例。

这个回答要真实：

> 我不会说这个框架已经很完整，它目前更像一个能跑通主流程、能体现自动化框架设计思路的练习项目。面试时我会重点讲清楚它已经实现了什么，以及我知道后续应该怎么补齐。

## 接口测试补充题

### 1. 如果给你一个接口测试，你怎么写用例？

> 我会先看接口文档，确认请求方式、URL、请求头、入参、返回字段和业务规则。然后从正常场景、异常参数、边界值、权限鉴权、业务状态、数据落库这几个角度设计用例。

可以这样展开：

- 正常场景：参数正确，接口返回成功。
- 必填参数：缺少必填字段是否提示。
- 参数类型：字段类型错误、格式错误。
- 边界值：数量、金额、长度、分页参数。
- 权限鉴权：不传 token、token 失效、越权访问。
- 业务规则：库存不足、重复提交、状态不允许操作。
- 数据校验：接口返回、页面展示、数据库结果是否一致。

### 2. 如果接口返回成功，但页面数据不对，你怎么排查？

> 我会先判断问题出在接口、前端还是数据层。第一步看前端实际请求的接口参数是否正确；第二步看接口响应数据是否正确；第三步如果接口响应正确但页面展示不对，就看前端字段映射、缓存或渲染逻辑；如果接口响应本身不对，就结合 SQL 查库和日志继续定位。

## Python 基础补充题

防止面试官根据简历里的 `Python + pytest + requests` 追问基础。

### 1. Python 常用库你用过哪些？

> 我在接口测试里常用的是 `requests`、`pytest`、`yaml`、`jsonpath`、`re`、`os`、`time` 这些。`requests` 用来发送 HTTP 请求，`pytest` 用来组织和执行测试用例，`yaml` 用来读取接口测试数据，`jsonpath` 用来从响应里提取字段，`re` 可以做简单的字符串匹配，`os` 用来处理路径，`time` 可以记录接口耗时。

结合我的框架：

> 我的 `E:\pytest_auto_api_my` 里主要用到了 `pytest`、`requests`、`PyYAML`、`jsonpath`，分别对应用例执行、请求发送、YAML 数据读取和响应字段提取。

### 2. Python 中如何定义一个函数？

> Python 用 `def` 定义函数，函数可以接收参数，也可以通过 `return` 返回结果。

```python
def add(a, b):
    return a + b
```

结合接口测试：

> 在接口自动化里，我会把重复逻辑封装成函数或类方法，比如发送请求、读取 YAML、执行断言、提取字段，这样测试代码不会到处重复。

### 3. Python 的装饰器是什么？

> 装饰器本质上是在不修改原函数代码的情况下，给函数增加额外功能。它通常写在函数上方，用 `@装饰器名` 的形式。比如在测试里，`pytest.mark.parametrize` 可以给测试函数做参数化，`pytest.fixture` 可以把一个函数声明成前置/后置夹具。

简单理解：

```python
@pytest.mark.parametrize("username,password", [("test", "123456")])
def test_login(username, password):
    pass
```

面试说法：

> 我对装饰器的理解是，它可以把一些公共能力加到函数外层，比如参数化、前置准备、权限校验、日志记录等。接口自动化里最常见的就是 pytest 的参数化和 fixture。

结合我的框架：

> 我的项目里会用 pytest 的 fixture 做上下文清理、登录初始化、环境切换；这些虽然不需要我手写复杂装饰器，但要理解 `@pytest.fixture` 这种写法是在给函数增加 pytest 识别和管理的能力。

### 4. `None` 是什么？

> `None` 表示空值或者没有返回值。比如函数没有写 `return`，默认返回 `None`；字典里取不到某个 key，也可能得到 `None`。接口测试里要注意判断字段是否为 `None`，比如 token 没提取到，就不能继续执行后续依赖接口。

结合我的框架：

> 我在上下文取 token、product_id 这类变量时，如果取到的是 `None`，就说明上游接口没有正确提取，后面的接口很可能会失败。

### 5. list、tuple、dict 有什么区别？

> `list` 是列表，可以增删改，适合存放一组可变数据；`tuple` 是元组，定义后一般不修改；`dict` 是字典，用 key-value 存数据，接口返回的 JSON 在 Python 里经常会被解析成 dict。

结合接口测试：

> 接口响应 JSON 通常是 dict 或 list。比如响应里的 `data` 可能是 dict，商品列表可能是 list，做断言和字段提取时要先看清楚响应结构。

### 6. 可变对象和不可变对象有什么区别？

> 可变对象是内容可以改，内存地址一般不变，比如 `list`、`dict`、`set`；不可变对象是内容不能直接改，比如 `str`、`int`、`tuple`。测试里最常见的是 dict 和 list，因为接口参数和响应数据经常用它们表示。

面试里可以简单说：

> 我不会把这个讲得太深，但知道 list、dict 可以修改，字符串修改后本质上会生成新的字符串。

### 7. 字符串替换怎么做？

> Python 里可以用 `replace` 做简单替换。

```python
text = "hello B"
new_text = text.replace("B", "A")
```

结合接口测试：

> 在接口自动化里，类似 `${{token}}`、`${{product_id}}` 这种占位符，本质上也是把字符串里的变量标记替换成上下文里的真实值。

### 8. Python 如何处理异常？

> Python 用 `try...except` 处理异常。接口测试里常见异常有请求超时、连接失败、响应不是 JSON、字段提取失败等。不能让一个异常直接把问题隐藏掉，最好保留请求参数和响应信息，方便定位。

```python
try:
    result = response.json()
except ValueError:
    result = response.text
```

结合我的框架：

> 我的请求封装里会捕获 `requests.RequestException`，并返回请求 method、url、headers、params、body 等调试信息，这样接口失败时能知道到底发了什么请求。

### 9. requests 怎么发送 GET 和 POST？

> `requests.get()` 一般用于查询接口，参数常放 `params`；`requests.post()` 一般用于提交接口，JSON 请求体常放 `json`。

```python
import requests

resp = requests.get("http://example.com/api", params={"page": 1})
resp = requests.post("http://example.com/login", json={"username": "test", "password": "123456"})
```

结合我的框架：

> 我没有在每条用例里直接写 `requests.get()` 或 `requests.post()`，而是封装了统一的 `send_request`，根据 YAML 里的 method、url、params、json 来发送请求。

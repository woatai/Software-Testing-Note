# 连星面试总结

## 问题一：为什么开发转测试？

我并不是完全“转”，而是发现软件测试这个方向更适合我的性格和能力。
在做项目开发或课程实验时，我对“系统是否稳定、功能是否符合预期”这类问题特别敏感，也愿意去发现隐藏的细节问题。
在实习过程中接触了测试工作，发现自己在执行用例、分析日志、复现问题这块有优势，也享受“发现问题、推动解决”的过程，所以我更加坚定了从事软件测试工作的方向。

📌 **重点是不要贬低开发，要强调你“更擅长、也更热爱测试”这一套流程。**

## 问题二：测试工具的使用

在实习期间，我使用过以下几种常见的测试工具：

- **Logcat**：用于抓取安卓系统运行日志，帮助分析崩溃、异常等问题；
- **Monkey**：执行压力测试，观察系统是否出现ANR或闪退等情况；
- **Bugreport**：用于一次性收集系统的完整状态和日志，用于复杂问题分析；
- **截图录屏工具**：配合复现问题并反馈开发；
- 此外，我还会使用 **adb 命令**进行操作模拟、环境搭建等辅助测试。

📌 这里你的实习经验就是最大优势，**能说出“为什么用、怎么用、在哪用”就很加分**。

## 问题三：如果出问题了，我们在文档里面有什么，才能更好的描述问题（缺陷）呢？【重要】

一个完整的缺陷描述文档/缺陷单（bug单）应包含以下要素，方便开发快速理解与定位问题：

- **缺陷标题**：简洁描述问题核心（如“相机闪光灯无响应”）；
- **复现步骤**：分步骤列出从打开APP到触发问题的具体操作路径；
- **实际结果 vs 预期结果**：明确指出偏差；
- **设备信息**：包含型号、系统版本、APP版本；
- **截图/录屏 + 日志**：提供视觉和技术证据；
- **严重程度与优先级**：评估该问题对用户的影响；
- （有时还可补充环境、网络状态、是否首次出现等）

我的回答是，首先bug标题这块一定要严谨，会有，涉及对应的项目代号，归属哪个模块，bug发生描述，输入加动作，实际结果。

对应的版本地址，型号

如果是用例发现的话，注明对应的用例，

对应的bug严重等级，有分为阻塞，严重，一般，，

有对应的前置条件，操作步骤，预期结果，实际结果，备注，

## 问题四：接口测试，postman请求参数是什么？

在 Postman 中发送请求时，常见的请求参数包括：

- **请求方式**：GET / POST / PUT / DELETE；

- **请求URL**：目标接口地址，如 `http://api.xxx.com/user/login`；

- **请求头（Headers）**：比如设置 `Content-Type: application/json`，或携带 `Authorization: Token`；

- **请求体（Body）**：主要用于 POST 或 PUT，一般为 JSON 格式，如：

  ```
  json复制编辑{
    "username": "test",
    "password": "123456"
  }
  ```

- 有时还需要设置 Query 参数（?key=value）或 Path 参数（如 /user/{id}）

- 在测试中主要用于验证接口返回的数据与数据库是否一致。

# 6.10 面试总结

公司工作需要

+ 用python进行数据清洗
+ jmeter进行压测
+ 输出压力测试报告

`【1】Python 数据清洗`

重点掌握以下库和技能（建议练习实际数据）：

- `pandas`：读取 CSV、Excel，数据筛选、清洗、去重、合并。
- `re`：正则表达式，提取、清洗字符串（如手机号、邮箱、日期等格式）。
- `openpyxl` / `xlrd`：读写 Excel 报表。
- `json` / `csv` 模块：清洗结构化数据。

🔧 示例任务：读取一个 5000 条记录的 Excel 文件，清洗出符合条件的用户数据，并存为新文件。

`【2】JMeter 压力测试 & 报告输出`

重点掌握：

- **线程组设置**：并发用户数（Number of Threads）、循环次数（Loop Count）、Ramp-Up 时间。
- **取样器**：HTTP Request。
- **监听器**：
  - 聚合报告（Aggregate Report）
  - 查看结果树（View Results Tree）
  - 图形结果（Graph Results）
- **断言**：如响应码为 200，或响应时间小于 1000ms。

💡 **压力测试报告要包含**：

- TPS（每秒事务数）
- 平均响应时间
- 最大响应时间
- 错误率
- 成功率
- 结论建议（例如系统在并发用户超过多少时明显卡顿）

🔧 示例任务：对某个登录接口进行 1000 用户并发请求测试，输出聚合报告，并撰写一段总结说明。

## 问题一：客户对测试方式有自己认知，而你们的测试团队走了另一套思路，最后需要全部推翻，客户表达不满

第一步：

首先我会主动和客户沟通，表达我们的重视和歉意，明确我们会快速响应调整测试计划，并请客户进一步明确他们期望的测试方法或标准。避免客户继续情绪化是首要任务。

第二步：

我们会回顾双方之前的**沟通文档**和**需求说明书**，确认测试思路不一致是由于哪些信息理解偏差，还是缺乏评审机制导致的执行偏差。这样可以避免以后再出类似问题。

第三步：

我会联合项目组快速梳理客户需求，重写测试用例，安排资源重新开展测试。并设立快速反馈机制，测试结果及时同步客户，争取短时间内重建信任。

## 问题二：规避漏测，误测问题

我更倾向于通过流程和工具手段来规避漏测和用例执行错误的问题。

**测试管理平台**：强制要求用例关联需求、执行状态可追踪

**日报机制 + 仪表盘**：可视化进度和覆盖率，及时发现异常。

## 问题三：测试用例分配不均的情况

首先，我肯定不会马上下结论说不公平，这不理性。测试难度不同、每个人熟悉的模块不同，或者临时任务调整。

我会和分配用例的测试工程师沟通，了解具体分配依据。如果确实存在不均衡，提出自己的疑问，但目的是希望分配更加合理，不是抱怨。

当然，在当前测试任务中，我还是会尽最大努力按时保质完成工作。我认为在团队**合作**中，**理解**、**沟通**和**灵活**应对都很重要。”

## 问题四： 面对用例比较多的情况怎么处理

我会先通览一遍所有用例，判断哪些是核心功能，哪些是边缘功能，进行任务拆解并设置优先级。

我会给自己制定自己的一个测试计划，把测试任务划分，比如每天完成多少条，甚至细致到每小时完成多少条，而且我会预留回归验证和问题跟踪的时间，用对应的测试管理工具去记录进度。

如果在最后用例有风险的情况下，我会和对应的te反馈，看是否可以进行人力调配，从而保证用例质量。

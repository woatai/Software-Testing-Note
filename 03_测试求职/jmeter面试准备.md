# JMeter 面试准备

## 这份笔记怎么用

这份是软件测试岗的 JMeter 冲刺版。目标不是把自己包装成性能测试专家，而是能讲清楚：

- 会用 JMeter 做接口级基础压测。
- 会配置线程组、HTTP 请求、请求头、断言和聚合报告。
- 会看响应时间、错误率、吞吐量。
- 会通过查看结果树定位失败原因。
- 知道正式压测前要确认环境、数据隔离和监控指标。

一句话定位：

> JMeter 我主要掌握接口级基础压测。比如登录接口和商品列表接口，我会先用 Apifox 确认接口能正常请求，再在 JMeter 里配置线程组、HTTP 请求默认值、请求头、状态码断言、业务断言和聚合报告，通过错误率、响应时间和吞吐量判断接口在小并发下是否稳定。

## 我的实操场景

目前可以结合这两个接口讲：

```text
登录接口：
POST http://39.102.52.234/api/login
断言：状态码 200，响应里包含 token。

商品列表接口：
GET http://39.102.52.234/api/home/products?page=1&limit=14
断言：状态码 200，响应里包含 success、list、store_name。
```

测试计划结构可以这样讲：

```text
测试计划
├── HTTP请求默认值：协议、IP、端口、/api
├── HTTP信息头管理器：Content-Type
├── 登录接口压测
│   ├── 登录请求
│   ├── 状态码断言
│   ├── 业务断言-token
│   ├── 查看结果树
│   └── 聚合报告
└── 商品列表压测
    ├── 商品列表请求
    ├── 状态码断言
    ├── 业务断言-list/store_name
    ├── 查看结果树
    └── 聚合报告
```

## 压测基本流程

1. 先在 Apifox 里确认接口能调通，记录请求方式、路径、参数、请求头和返回字段。
2. 在 JMeter 里创建线程组，设置线程数、Ramp-Up 和循环次数。
3. 用 HTTP 请求默认值维护公共配置，比如协议、服务器 IP、端口和 `/api`。
4. 用 HTTP 信息头管理器维护公共请求头，比如 `Content-Type: application/json`。
5. 添加 HTTP 请求，先用 1 个线程、1 次循环确认能跑通。
6. 添加状态码断言和业务断言，避免只看 HTTP 200。
7. 小并发逐步加压，比如 5、10、20 线程。
8. 查看聚合报告，重点看平均响应时间、最大响应时间、错误率和吞吐量。
9. 如果失败，回到查看结果树看请求参数、响应体和断言失败原因。

## 必背高频题

### 1. 你用 JMeter 做过什么？

思路：

- 接口级基础压测
- 登录和商品列表
- 配置项
- 看报告

回答：

> 我用 JMeter 做过接口级基础压测，主要是登录接口和商品列表接口。流程是先在 Apifox 里确认接口能调通，然后在 JMeter 里配置线程组、HTTP 请求默认值、HTTP 信息头管理器、请求参数、状态码断言和业务断言，最后通过查看结果树定位失败原因，通过聚合报告看平均响应时间、最大响应时间、错误率和吞吐量。

### 2. JMeter 做接口压测的流程是什么？

思路：

- Apifox 先确认
- JMeter 搭脚本
- 单线程跑通
- 加断言
- 逐步加压
- 看报告

回答：

> 我一般先用 Apifox 确认接口请求方式、路径、参数和返回字段。然后在 JMeter 里创建线程组，配置 HTTP 请求默认值、请求头和 HTTP 请求。单接口先用 1 个线程跑通，再加状态码断言和业务断言，确认没有问题后再逐步增加线程数。最后看查看结果树和聚合报告，重点关注错误率、响应时间和吞吐量。

### 3. 线程数、Ramp-Up、循环次数是什么意思？

思路：

- 线程数：模拟用户数
- Ramp-Up：启动这些用户的时间
- 循环次数：每个用户执行几轮

回答：

> 线程数可以理解为模拟多少个并发用户，Ramp-Up 是这些用户在多少秒内启动完成，循环次数是每个线程执行几轮。比如设置 10 个线程、Ramp-Up 10 秒、循环 3 次，总请求数大概是 30 次，而且不是一下子全部打上去，而是在 10 秒内逐步启动。

### 4. 你为什么用 HTTP 请求默认值？

思路：

- 公共配置抽取
- 减少重复
- 环境切换方便

回答：

> 我会用 HTTP 请求默认值维护公共配置，比如协议、服务器 IP、端口和 `/api` 前缀。这样单个接口只需要写自己的路径，比如登录写 `/login`，商品列表写 `/home/products`。如果环境地址变了，只需要改一处，和 Apifox 里配置环境变量的思路类似。

### 5. 你怎么配置请求头？

思路：

- HTTP 信息头管理器
- Content-Type
- token

回答：

> 我会用 HTTP 信息头管理器统一配置请求头。比如 JSON 请求会加 `Content-Type: application/json`。如果接口需要登录态，就在请求头里加 token，比如 `Authorization: Bearer ${token}`。这样不用每个接口重复配置请求头，后续维护也方便。

### 6. 你怎么做 JMeter 断言？

思路：

- 状态码断言
- 业务断言
- 不只看 200

回答：

> 我不会只判断 HTTP 200，一般会加两类断言。第一类是状态码断言，比如响应码等于 200。第二类是业务断言，比如登录接口响应里必须包含 `token`，商品列表接口响应里必须包含 `success`、`list`、`store_name`。这样可以避免 HTTP 请求成功但业务实际失败的情况。

### 7. 查看结果树主要看什么？

思路：

- 请求参数
- 响应码
- 响应体
- 断言失败原因

回答：

> 查看结果树主要用于调试和定位问题。我会看请求路径、请求参数、请求头是否正确，再看响应码和响应体。如果请求是红色失败，我会重点看是状态码失败，还是业务断言失败。比如响应码是 200 但没有 token，就说明 HTTP 成功了，但登录业务没有真正成功。

### 8. 聚合报告怎么看？

思路：

- 样本数
- 响应时间
- 错误率
- 吞吐量

回答：

> 聚合报告我主要看四类指标：样本数、响应时间、错误率和吞吐量。响应时间会看平均值、最大值、90% 和 95% 分位；错误率看接口有没有失败；吞吐量看每秒能处理多少请求。比如我登录接口跑过 30 次请求，平均响应大概 220ms，错误率 0%，说明小并发下接口比较稳定。

### 9. 如果聚合报告里错误率很高，你怎么排查？

思路：

- 先看失败请求
- 区分 HTTP 失败和业务断言失败
- 检查参数、请求头、账号和并发

回答：

> 我会先到查看结果树里找到失败请求，看响应码、响应体和断言结果。如果响应码不是 200，比如 500、502、403，可能是服务端异常、权限问题或网关问题。如果响应码是 200，但业务断言失败，比如没有返回 token，那就是业务失败或断言字段不稳定。然后我会检查参数、请求头、账号数据，再降低并发重新验证。

### 10. 为什么不能一上来就把并发压很高？

思路：

- 先验证脚本
- 保护测试环境
- 避免数据污染
- 逐步加压

回答：

> 我不会一开始就猛压，尤其是自己的练习服务器或测试环境。因为脚本、参数、断言还没确认时，直接加大并发可能把服务打挂，也可能产生脏数据。我的做法是先 1 个线程跑通，再用 5、10、20 这种方式逐步加压，同时观察错误率和响应时间变化。正式压测前还要确认压测环境、测试数据和监控指标。

## 加分追问题

### 11. 单接口压测和链路压测有什么区别？

思路：

- 单接口看某个接口
- 链路压测模拟真实业务流程
- 链路需要参数传递

回答：

> 单接口压测是单独压一个接口，比如登录接口或商品列表接口，适合先验证接口本身是否稳定。链路压测是按照真实业务流程串起来，比如登录、商品列表、商品详情、确认订单、提交订单。链路压测里上一个接口返回的数据要传给下一个接口，比如 token、商品 ID、订单 key，一般用 JSON 提取器提取变量，再在后续请求中引用。

### 12. JMeter 里怎么做接口关联？

思路：

- 上个接口提取
- 下个接口引用
- token、商品 ID、订单号

回答：

> 接口关联就是上一个接口返回的数据要给后面的接口用。比如登录接口返回 token，后续个人中心或下单接口要带 token；商品列表返回商品 ID，商品详情接口要用商品 ID。JMeter 里可以用 JSON 提取器把字段提取成变量，比如 `${token}`、`${product_id}`，再在后续请求里引用。

### 13. CSV Data Set Config 是做什么的？

思路：

- 参数化
- 多组测试数据
- 登录账号、商品 ID

回答：

> CSV Data Set Config 主要用于参数化。比如登录接口如果要用多组账号压测，可以把账号和密码放到 CSV 文件里，然后在请求体里用 `${account}`、`${password}` 引用。这样不用在 JMeter 里写死一组数据，也更接近多个用户请求的场景。

### 14. 如果面试官问性能瓶颈怎么定位，你怎么答？

思路：

- 测试岗口径
- 先从报告看现象
- 再结合日志、数据库和监控

回答：

> 我目前主要掌握接口级基础压测。遇到响应慢或错误率高，我会先从 JMeter 报告判断是整体变慢还是部分请求失败，再看查看结果树里的失败响应。如果是公司正式环境，我会结合服务日志、数据库慢 SQL、CPU、内存、网络和接口错误日志一起排查。作为测试，我会先把现象、并发配置、错误率、响应时间和复现步骤整理清楚，再推动开发一起定位。

## 不要主动深讲

下面这些了解即可，除非面试官主动问：

- 分布式压测。
- 非 GUI 模式命令执行。
- 复杂性能调优。
- JVM 调优。
- 服务端底层瓶颈分析。
- 压测平台建设。

你的岗位定位是：

> 功能测试和接口测试为主，JMeter 是接口性能测试的基础能力，不要把自己讲成专门性能测试岗。

## 最后速背

面试官问 JMeter，可以用这一段兜底：

> JMeter 我主要用于接口基础压测。我会先用 Apifox 确认接口可用，再在 JMeter 里配置线程组、HTTP 请求默认值、请求头、请求参数、状态码断言和业务断言。执行后通过查看结果树定位失败原因，通过聚合报告看平均响应时间、最大响应时间、错误率和吞吐量。我做过登录接口和商品列表接口的小并发压测，能判断接口在基础并发下是否稳定。正式压测时，我会先确认环境、数据、监控和压测目标，再逐步加压。
